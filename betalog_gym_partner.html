The opportunity is a **B2B2C platform** — a product that sells to gyms (B2B) who then deliver it to their members (B2C). **B2B2C explained simply:** You sell to the gym. The gym gives it to their members. The climber is the end user. The gym is the paying customer. You serve both but only one pays you directly. This model works because: - Climbers get a free, genuinely useful app — low barrier to adoption - Gyms get a member engagement and retention tool they don't have to build themselves - You get recurring monthly revenue from gyms rather than trying to charge thousands of individual climbers small amounts - The gym does your marketing for you by telling their members about it ### The gap in the market Most indoor climbing gyms have no training tool to offer members beyond generic fitness apps that know nothing about climbing. Vertical Life and The Crag cover outdoor route databases. Nobody has nailed the **training + gym integration + social layer** combination for indoor climbing. That is the space BetaLog is building into. There are roughly 600–700 indoor climbing walls in the UK. The majority are independent or small operators with no member-facing digital product. ### Revenue model Gyms pay a monthly subscription for the admin dashboard and member tools. Members always use the app free. Illustrative tiers (not fixed — first partner sets the terms): | Tier | Price | What's included | |---|---|---| | Free | £0/month | Gym profile page, logo, website link | | Partner | ~£99/month | Route board, member auto-onboarding, coaching profiles, booking links, challenges | | Pro | ~£249/month | Everything + analytics, AI coach with gym context, comp features | Revenue at scale (illustrative): | Gyms on platform | Avg fee | Annual revenue | |---|---|---| | 5 | £99 | ~£6k | | 20 | £150 | ~£36k | | 60 | £150 | ~£108k | | 65 | £180 | ~£140k | This is purely gym subscription revenue — does not include individual climber premium features, white-labelling, or an eventual sale of the business. ### Exit options **A gym buys it and runs it as a SaaS business** A gym owner acquires BetaLog — the codebase, brand, user base — and sells access to other gyms. They own the platform their competitors pay to use. Rock Gym Pro, the dominant gym management system, started exactly this way — built by a climber for one gym, expanded from there. **Sell to a larger player** Once the platform has users and paying gyms, it becomes attractive to: - Gym management companies (Rock Gym Pro, Mindbody, Onsight Pro) wanting a member-facing training layer on top of their back-office systems - Climbing brands (Lattice, Tension, Moon) wanting a direct digital channel to climbers - Fitness tech companies expanding into vertical sports Acquisition value is driven by active user count, recurring gym revenue, and quality of member training data. All three grow together. **Stay independent** Take on a gym partner, use that relationship to build, keep full upside. ### The first gym partner A close friend owns **Redpoint**, a climbing gym. This is the intended first partner — informal pilot, no formal contract needed initially. The goal is: 1. Share the app with their members and get honest usage feedback 2. Get a gym operator's perspective on what's missing 3. Test the RGP integration with a real gym setup 4. Use Redpoint as the reference case when approaching other gyms **Note on naming:** The app is called BetaLog (not Redpoint) intentionally. An app named after one gym can never be sold to another gym. BetaLog is the independent platform — Redpoint is the founding partner. "Powered by BetaLog" is the relationship. This also means if the product is eventually sold or expanded, it isn't tied to one gym's identity. --- ## Protecting Yourself as Founder This section exists because these conversations are easier to have before there is anything worth arguing about. ### Own the infrastructure — personally - **GitHub repo:** stays in your name, others added as collaborators not owners - **Domain (`betalog.co.uk`):** registered personally, not under a company or partner - **Firebase project:** set up under your Google account, grant access to others If the relationship with any partner sours, you can walk away with the product intact. ### If you bring in a developer partner You will likely need one eventually. Most serious climbing gyms run it. Understanding what it can and can't do is important before building around it. ### What RGP actually knows **Has:** - Member name, email, phone - Membership type and status (active, frozen, cancelled) - Check-in timestamps (date/time they tapped in) - Bookings (classes, coaching sessions booked) - Customer tags (e.g. youth team, competition squad, beginner course) - Invoice/payment history **Does NOT have:** - What the member climbed - Their grades or sends - How long they stayed - Any climbing performance data The RGP integration is about **identity and visit data** — not climbing data. Everything climbing-specific still comes from BetaLog itself. RGP is the door in, not the engine. ### How the integration works technically RGP has a read-only API and a Zapier integration. Zapier fires events (new member, check-in, booking) which POST to any webhook URL — including a Firebase Cloud Function. ``` Member joins gym (or checks in) → RGP fires event to Zapier → Zapier POSTs to your Firebase Cloud Function URL → Firebase creates/updates user account → Welcome email sent to member ``` The only real dev work is one Cloud Function (~30 lines): ```javascript exports.rgpWebhook = functions.https.onRequest(async (req, res) => { const { email, name, gymId } = req.body // Check if user exists in Firebase Auth // If not, create account and send welcome email // Create/update Firestore user doc with gymId res.status(200).send('ok'); }); ``` Note: Cloud Functions that make outbound network calls require Firebase Blaze (pay-as-you-go) plan. Cost at small scale is negligible — typically under £5/month. ### What the RGP integration realistically enables 1. **Auto account creation** — member joins gym → BetaLog account provisioned automatically 2. **Membership verification** — app knows if membership is active, frozen, or lapsed 3. **Check-in → session stub** — tap in at gym → blank session created in app pre-filled with date and gym 4. **Booking nudges** — AI coach can surface coaching suggestions linked directly to gym's booking page 5. **Customer tag → personalisation** — youth team, competition squad etc. flow through to app experience ### Legal requirements for this integration - The gym must have opt-in language in their membership terms: *"your data may be shared with BetaLog to provide training tools"* - A data processing agreement (DPA) between you and the gym before any data flows - GDPR compliance on your side for storage and use of member data - The gym is responsible for member consent; you are responsible for what you do with the data once you have it - Template DPAs are available online — standard practice for this kind of arrangement --- ## Technical Roadmap ### Where the code is now Single file `index.html` — ~7,600 lines. All data in localStorage. No accounts, no backend, no database. Fine for personal use. For a PoC with real users, needs: 1. Code split into separate files 2. Cloud database (Firebase Firestore) 3. User accounts (Firebase Auth) ### The correct order Do not add Firebase to a 7,600-line single file — the result is hard to debug and impossible to hand to anyone else. Split first. ``` Step 1 — Split the file into separate JS/CSS files Step 2 — Create a clean data layer (the "adapter") Step 3 — Set up Firebase project Step 4 — Swap the adapter to use Firebase Step 5 — Add login UI Step 6 — Point betalog.co.uk at GitHub Pages Step 7 — Test end to end ``` Every step is self-contained. The app works after every step. --- ### Step 1 — Split into Separate Files **Target structure:** ``` / ├── index.html ← HTML skeleton only (~300 lines) ├── css/ │ └── app.css ← all styles └── js/ ├── data.js ← loadData(), save(), and later Firebase sync ├── dashboard.js ← dashboard rendering and charts ├── log.js ← session logging (gym, climb, hangboard) ├── train.js ← exercise library, routines, schedule ├── coach.js ← AI coach (Groq API calls) └── app.js ← nav, streak, badges, init ``` **How (GitHub web editor — no computer needed):** 1. Go to repo → Add file → Create new file 2. Type `js/data.js` as filename (GitHub auto-creates the folder) 3. Copy relevant functions from `index.html` 4. Commit 5. Replace that block in `index.html` with `` 6. Open live site — if it works, the split was clean. If not: F12 → Console → fix the error 7. Repeat per file **Test after every single split. Fix before moving on.** **Order (safest first):** 1. `css/app.css` — styles only, nothing can break 2. `js/data.js` — loadData() and save() 3. `js/coach.js` — Groq AI functions (self-contained) 4. `js/dashboard.js` — dashboard render functions 5. `js/log.js` — logging panel functions 6. `js/train.js` — exercises, routines, schedule 7. `js/app.js` — everything else --- ### Step 2 — Clean Data Adapter **Current (talks directly to localStorage):** ```js function save(key, val) { localStorage.setItem(key, JSON.stringify(val)); } ``` **New (adapter — swap internals in Step 4 without touching anything else):** ```js var Storage = { save: function(key, val) { try { localStorage.setItem(key, JSON.stringify(val)); } catch(e) { alert('Storage full — export a backup to avoid losing data.'); } // Firebase write added here in Step 4 — nothing else in the app changes }, load: function(key, fallback) { try { var val = localStorage.getItem(key); return val ? JSON.parse(val) : fallback; } catch(e) { return fallback; } } }; ``` **Why this matters:** In Step 4 you change one file (`data.js`) instead of hunting through 7,600 lines for every localStorage call. --- ### Step 3 — Set Up Firebase 1. Check F12 console for Firebase errors. **"App breaks after splitting a file"** F12 → Console tells you exactly which function is undefined.
