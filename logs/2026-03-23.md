## Design decisions

**Session logging ‚Äî no live set tracking**
Decided against real-time set-by-set tracking during a workout. Instead:
- Gym log defaults all sets to "done" with the routine's target reps/weight on submit
- User can adjust individual sets (reps, weight) before hitting Save if actuals differed
- Keeps the log flow fast ‚Äî in and out, not a workout companion app
- Build order: gym log + history first, climb log second, hangboard Step 4

## Gotchas
- `npm run dev` must be run from inside `betalog-react/`, not the parent folder
- After taxonomy overhaul, existing localStorage `il_exercises` seeded under old category values won't auto-update (category field already set ‚Äî migration skips it). Clear `il_exercises` to re-seed.
- Merge-based seeding: `seedDefaultExercises()` only adds IDs not already present ‚Äî safe for existing users with custom exercises

## Build order decision ‚Äî History before climb/hangboard logging

Revised build order agreed:
1. ~~Step 3a ‚Äî Gym logging~~ ‚úÖ
2. **Step 3b ‚Äî History** (next)
3. Step 3c ‚Äî Climb log
4. Step 4 ‚Äî Hangboard timer + log

Rationale: gym logging already saves real data but there's no way to view it. History closes the loop and is a pure read ‚Äî quick to build and immediately validates the data model. Climb + hangboard logging follow after.

**Key constraint for History:** build it to handle all three session types (gym, climb, hangboard) from day one ‚Äî not gym-only. Climb and hangboard sessions have different shapes (`session.climbs[]`, `session.hangGrips[]`) and the card design + detail view should accommodate all of them before any are wired up.
