# BetaLog — Data Model

This is the single source of truth for all data structures in BetaLog.
All code — React components, storage adapters, and (future) Firestore rules — must conform to this model.
Update this document when any data structure changes.

Last updated: 2026-03-25

---

## Storage keys (localStorage Phase 1 / Firestore Phase 2)

| Key | Type | Description |
|---|---|---|
| `il_sessions` | Session[] | All logged sessions |
| `il_exercises` | Exercise[] | User's exercise library |
| `il_routines` | Routine[] | Gym and hangboard routines |
| `il_schedule` | Schedule \| null | Weekly training schedule |
| `il_weightLog` | WeightEntry[] | Bodyweight log |
| `il_athleteProfile` | AthleteProfile | User profile |
| `il_badges` | string[] | Earned badge IDs |
| `il_groq_key` | string | User-supplied Groq API key (raw string) |

In Firestore (Phase 2), all collections live under `users/{userId}/` except gym/route
data which is shared under `gyms/{gymId}/`.

## Session

The central record. One session = one training visit.

```ts
interface Session {
  id:          string          // Date.now().toString() or uuid
  date:        string          // ISO date "YYYY-MM-DD" — when the workout happened (user-editable)
  type:        "gym" | "climb" | "hangboard"
  discipline:  "boulder" | "lead" | "toprope" | null  // set when type === "climb", else null
  routineId:   string | null   // id of the Routine used; null for ad-hoc / exercise-only sessions
  routineName: string | null   // denormalised routine name — survives routine deletion/rename
  difficulty:  1 | 2 | 3 | 4 | 5  // perceived effort, 1=easy 5=max
  notes:       string          // free text, may be empty string
  exercises:   SessionExercise[]  // populated when type === "gym", else []
  climbs:      Climb[]            // populated when type === "climb", else []
  hangGrips:   HangGrip[]         // populated when type === "hangboard", else []
  createdAt:   string          // ISO datetime, set on creation, never changed
  updatedAt:   string          // ISO datetime, bumped on every edit
}
```

### Session field matrix — what each session type uses

Every session has the same shape. Fields not relevant to a type are set to `null` or `[]`.
This table is the definitive reference for dashboard queries, history display, and edit sheets.

| Field | `gym` | `climb` | `hangboard` | Notes |
|---|---|---|---|---|
| `id` | uuid | uuid | uuid | auto-generated by `useSessions` |
| `date` | YYYY-MM-DD | YYYY-MM-DD | YYYY-MM-DD | user-editable via date picker |
| `type` | `"gym"` | `"climb"` | `"hangboard"` | immutable after creation |
| `discipline` | `null` | boulder / lead / toprope | `null` | derived from climbs; `null` if mixed |
| `routineId` | routine id or `null` | `null` | routine id or `null` | `null` for ad-hoc / free hang |
| `routineName` | routine name or `null` | `null` | routine name or `null` | `null` for free hang |
| `difficulty` | 1–5 | 1–5 | 1–5 | required on all types |
| `notes` | string | string | string | optional, may be `""` |
| `exercises[]` | **populated** | `[]` | `[]` | `SessionExercise` objects |
| `climbs[]` | `[]` | **populated** | `[]` | `Climb` objects, oldest-first |
| `hangGrips[]` | `[]` | `[]` | **populated** | `HangGrip` objects |
| `createdAt` | ISO datetime | ISO datetime | ISO datetime | set once by `useSessions.addSession` |
| `updatedAt` | ISO datetime | ISO datetime | ISO datetime | bumped by `useSessions.updateSession` |

### `date` vs `createdAt` vs `updatedAt`

- **`date`** — the day the workout happened. User-editable (e.g. back-dating yesterday's session). This is what history groups by and what the dashboard reads.
- **`createdAt`** — when the record was first saved. Never changes. Used as a sort tiebreaker when two sessions share the same `date`.
- **`updatedAt`** — last edit timestamp. Bumped automatically by `updateSession()`. Will be used for Firebase sync conflict resolution.

### Migration note
Old sessions used `type: "boulder" | "lead" | "toprope"` directly (no separate
`discipline` field) and `hangProtocol` (single object) instead of `hangGrips` (array).
The storage layer must handle both shapes when reading. When writing, always use the
new shape. A one-time migration function should be run on first load of the React app.

## Climb

One climb attempt within a session. Sessions of type `"climb"` contain an array of these.

```ts
interface Climb {
  id:          string          // uuid, unique within session
  grade:       string          // "V4", "6b+", "7a", etc. — raw string as entered
  gradeSystem: "v" | "french" | "yds" | "uk_trad"  // grade scale used
  discipline:  "boulder" | "lead" | "toprope"  // always set — matches parent session discipline for standalone logs
  outcome:     "flashed" | "sent" | "attempt" | "project"
  attempts:    number          // number of attempts this session, minimum 1
  // Gym route link — all null for standalone (non-gym) logs
  routeId:     string | null
  gymId:       string | null
  centreId:    string | null
}
```

### Notes
- `gradeSystem` is derived from discipline in most cases (boulder → "v", lead/toprope → "french")
  but stored explicitly to allow user override (e.g. UK trad, YDS).
- `outcome: "fell"` from the old app is folded into `"attempt"` — use `attempts` count instead.
- `attempts` defaults to 1. A flash is always `attempts: 1`.

## SessionExercise

One exercise entry within a gym session. Not the library definition — a logged instance.

```ts
interface SessionExercise {
  exerciseId:   string          // reference to Exercise.id in the library
  name:         string          // denormalised from library for display (no join needed)
  trackingType: "reps" | "time" // determines whether set values are rep counts or seconds
  sets:         ExerciseSet[]
}

interface ExerciseSet {
  reps:    number
  weight:  number              // kg, 0 for bodyweight exercises
  rir:     number | null       // Reps in Reserve, null if not tracked
  done:    boolean             // ticked off during the session
}
```

## HangGrip

One grip block within a hangboard session. Sessions of type `"hangboard"` contain an
array of these — one entry per grip type/protocol block.

```ts
interface HangGrip {
  id:          string          // uuid
  fingers:     string          // e.g. "4 Finger", "Front 2", "Mono Index"
  gripType:    GripType        // canonical grip type, e.g. "half-crimp"
  grip:        GripType        // legacy alias of gripType — kept for storage compat
  edgeSize:    string          // e.g. "20mm", "Sloper", "Jug", or "" for unspecified
  gripName:    string          // human-readable label, e.g. "4 Finger · Half Crimp · 20mm"
  activeSecs:  number          // hang duration per rep in seconds
  restSecs:    number          // rest between reps in seconds
  setRest:     number          // rest between sets in seconds
  reps:        number
  sets:        number
  weightMode:  "bodyweight" | "added" | "assisted"
  weightKg:    number          // kg added or assisted, 0 if bodyweight
}
```

### HangGrip field reference

| Field | Purpose | Example |
|---|---|---|
| `fingers` | Which fingers are on the edge | `"4 Finger"`, `"Front 2"`, `"Mono Index"` |
| `gripType` | Grip position (canonical) | `"half-crimp"`, `"open-hand"`, `"full-crimp"` |
| `grip` | Legacy alias of `gripType` | Same values — always kept in sync |
| `edgeSize` | Edge depth or hold type | `"20mm"`, `"Sloper"`, `"Jug"`, `""` |
| `gripName` | Display label (derived) | `"4 Finger · Half Crimp · 20mm"` |

### Migration note
Old sessions used `hangProtocol` (single object) with fields `onSecs`/`offSecs` and
`weightDir`/`weightPct`. Map these on read:
- `onSecs` → `activeSecs`
- `offSecs` → `restSecs`
- `weightDir: "added"` + `weight` → `weightMode: "added"`, `weightKg: weight`
- `weightDir: "none"` → `weightMode: "bodyweight"`, `weightKg: 0`
- Wrap the single object in an array as `hangGrips: [proto]`

## Exercise (library entry)

The user's reusable exercise definitions. Not tied to any session.

```ts
interface Exercise {
  id:              string
  name:            string
  category:        "chest" | "back" | "shoulders" | "arms" | "legs" | "core" | "mobility" | "cardio" | "other"
  // category = what muscle group it trains — drives the filter chips in the UI
  // chest: pressing movements targeting pecs
  // back: rows, pull-ups, lat work
  // shoulders: overhead press, raises, rear delt isolation
  // arms: bicep/tricep isolation
  // legs: quads, hamstrings, glutes, calves
  // core: abs, obliques, lower back stability
  // mobility: stretching, flexibility, rehab, prehab
  // cardio: heart rate / conditioning work
  // other: catch-all

  movementPattern: "push" | "pull" | "hinge" | "squat" | "carry" | "rotation" | "isometric" | null
  // movementPattern = biomechanical classification — used by AI coach for program balance
  // push: pressing away from body (bench, OHP, dips)
  // pull: pulling toward body (rows, pull-ups, curls)
  // hinge: hip hinge (deadlift, RDL, glute bridge)
  // squat: knee-dominant lower body (squat, lunge, step-up)
  // carry: locomotion under load (farmer carry)
  // rotation: rotational core (cable woodchop, russian twist)
  // isometric: static holds (plank, dead hang, wall sit)
  // null: mobility / rehab work where pattern doesn't apply

  equipment:       "bw" | "db" | "bb" | "kb" | "band" | "cable" | "machine" | "other" | null
  muscles:         string          // e.g. "Lats, biceps, rear delts" — free text
  notes:           string          // coaching notes, may be empty
  ytSearch:        string          // custom YouTube search query; blank = "how to {name} form tutorial"
  isFavourite:     boolean         // pinned to top of exercise list
  trackingType:    "reps" | "time"  // "reps" = rep counter; "time" = countdown timer
  defaultSets:     number
  defaultReps:     number          // used when trackingType === "reps"
  defaultDuration: number          // seconds per set, used when trackingType === "time"
  defaultRest:     number          // seconds between sets
  defaultWeight:   number          // kg offset; 0 = bodyweight, positive = added, negative = assisted
  createdAt:       string          // ISO datetime
  updatedAt:       string          // ISO datetime
}
```

### Migration note
Old exercises used `mp` for movement pattern and `cat` for category. On read:
- `mp` → `movementPattern` (rename). Map `mp: "shoulder"` to the appropriate pattern: `"push"` for pressing movements, `"pull"` for rear-delt/row movements.
- `cat` → `category` (rename + remap): `"bodyweight"` / `"weights"` → derived from `mp`; `"pull"` → `"back"`; `"rehab"` → `"mobility"`.
- Old `cat` values (`"push"`, `"pull"`, `"hinge"`, `"squat"`, `"core"`, `"shoulder"`) do not match new `category` values (`"chest"`, `"back"`, `"shoulders"`, etc.) — migration maps by movement pattern only as a best effort; manual correction may be needed for some exercises.
Default exercises shipped with the app have `id` values prefixed `e` (e.g. `"e46"`).
User-created exercises use `Date.now().toString()` IDs — preserve these.

## Routine

A saved workout template. Used for both gym routines and hangboard routines.
Differentiated by the `type` field.

```ts
interface Routine {
  id:        string
  name:      string
  type:      "gym" | "hangboard"
  exercises: RoutineExercise[]   // populated when type === "gym", else []
  grips:     HangGrip[]          // populated when type === "hangboard", else []
  createdAt: string
  updatedAt: string
}

interface RoutineExercise {
  exerciseId:     string           // reference to Exercise.id
  name:           string           // denormalised
  trackingType:   "reps" | "time"  // inherited from Exercise on add
  targetSets:     number
  targetReps:     number           // used when trackingType === "reps"
  targetDuration: number           // seconds per set; used when trackingType === "time"
  targetRest:     number           // seconds between sets
  targetWeight:   number | null    // null = bodyweight or user decides on the day
  order:          number           // 0-indexed display order
}
```

## Schedule

The user's weekly training schedule. Stored as an array of up to 3 entries under `il_schedule`.
Managed in Plan → Profile → Schedule card.

```ts
type Schedule = ScheduleEntry[]  // max 3 entries

interface ScheduleEntry {
  id:          string          // uuid
  routineId:   string          // reference to Routine.id (gym or hangboard)
  routineName: string          // denormalised for display
  days:        number[]        // 1=Monday … 7=Sunday
}
```

### Migration note
Old format was a singleton `{ routineId, days, updatedAt }`. The `useSchedule` hook
auto-migrates this to the new array format on first read.

## WeightEntry

One bodyweight measurement. Created in Plan → Profile → Weight log.
Appears inline in the History feed (date-grouped alongside sessions) with edit/delete.

```ts
interface WeightEntry {
  id:     string
  date:   string                 // ISO date "YYYY-MM-DD"
  weight: number                 // kg
  note:   string | null
}
```

## AthleteProfile

User profile data. Used by the AI coach and for personalised stats.
Lives in the Plan → Profile tab.

```ts
interface AthleteProfile {
  name:         string
  heightCm:     number | null
  weightKg:     number | null    // baseline, not the same as weight log
  apeIndex:     number | null    // arm span minus height in cm (not in UI yet)
  climbingSince: string | null   // ISO date (not in UI yet)
  homeGym:      string | null    // free text (not in UI yet)
  goals:        string           // free text
  updatedAt:    string
}
```

### UI scope (current)
The Profile form exposes: `name`, `heightCm`, `weightKg`, `goals`.
The remaining fields (`apeIndex`, `climbingSince`, `homeGym`) are retained in the data model
for future use (e.g. AI coach context) but have no UI input currently.

## Firestore structure (Phase 2)

```
users/
  {userId}/
    profile/          (single doc: AthleteProfile)
    sessions/         (collection: Session[])
    exercises/        (collection: Exercise[])
    routines/         (collection: Routine[])
    weightLog/        (collection: WeightEntry[])
    schedule/         (single doc: Schedule)

gyms/
  {gymId}/
    name, location, etc.
    centres/
      {centreId}/
        name, wallMapUrl, etc.
        routes/
          {routeId}/
            grade, gradeSystem, colour, setter, setDate,
            photoUrl, stats: { totalLogs, flashes, sends, attempts }
```

## Grade systems reference

| System | Context | Example values |
|---|---|---|
| `"v"` | Bouldering (US) | V0, V1 … V17 |
| `"french"` | Sport / lead / toprope | 5a, 6b+, 7c, 8a |
| `"yds"` | Trad / US sport | 5.9, 5.10a, 5.12d |
| `"uk_trad"` | UK trad | Mod, VD, HVS, E3 |

When displaying grades in charts and comparisons, always filter to a single grade system.
Never sort or compare grades across systems.

## ID conventions

| Object | ID format |
|---|---|
| Sessions | `Date.now().toString()` (legacy) or UUID v4 |
| Exercises (default library) | `"e"` prefix + number, e.g. `"e46"` |
| Exercises (user-created) | `Date.now().toString()` |
| Routines | `Date.now().toString()` |
| Climbs | UUID v4 (new) |
| HangGrips | UUID v4 (new) |
| WeightEntries | UUID v4 (new) |

Moving forward, prefer UUID v4 for all new IDs. `Date.now()` IDs are retained for
backwards compatibility with existing data.
